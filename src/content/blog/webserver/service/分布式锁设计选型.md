# 设计原则
1、**互斥**：同一时间只能有一个线程持有同一把锁；

2、**可重入**：同一线程多次获取同一把锁，不应该阻塞；

3、**安全**：线程完成任务、崩溃、失败等清空下锁必须要释放；
- 要有**锁失效机制**：一般设计超时机制；

4、**性能**：分布式锁必然需要外部中间件，需要网络IO损耗；
- 网络IO损耗；
- 非阻塞特性：如果没有获取锁，最好直接返回，不要阻塞一直等待，这点需要根据业务场景分析；
  - 某些周期任务，没有获取锁就等待下一周期，不要阻塞；
  - 抢购，用户侧需要持续抢购，允许等待，这时可以阻塞等待；

5、**高可用**：不能因为中间件的不可用，导致服务不可用；

# 常见分布式锁实现方案

## MySQL
性能要求不高，且必须是非阻塞锁；

1、基于唯一索引插入实现

2、基于`for update`排他锁 + 事务
- 占用数据库连接

## Redis
性能高，核心命令：`setnx`
- 需要考虑高可用，集群下可能需要考虑红锁；
- 需要设置超时时间防止死锁，但是过期时间不好定，需要锁续期(看门狗机制)


## Zookeeper
